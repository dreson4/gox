# GOX Implementation Plan

## Architecture Overview

GOX has 5 major subsystems that build on each other:

```
┌─────────────────────────────────────────────────┐
│                   CLI (gox)                      │
│  init · run · build · dev · generate             │
├─────────────────────────────────────────────────┤
│              Native Code Generator               │
│  iOS project · Android project · auto-managed    │
├──────────────────────┬──────────────────────────┤
│    GOX Compiler      │     Runtime Library       │
│  .gox → .go          │  gox package + http       │
│  parser · transform  │  views · nav · state      │
├──────────────────────┴──────────────────────────┤
│              Platform Bridge                     │
│  Go ↔ UIKit (iOS) · Go ↔ Android Views          │
└─────────────────────────────────────────────────┘
```

## Phase 1: GOX Compiler (Parser + Transformer)
**Goal:** Parse `.gox` files and emit valid `.go` files.

This is the foundation — nothing else works without it. We start iOS-first.

### 1.1 — Lexer/Tokenizer
- Tokenize `.gox` files
- Handle two modes: **Go mode** (standard Go tokens) and **View mode** (JSX-like tokens)
- Go mode: pass through everything — package, import, type, func, var
- View mode starts at `view {` and ends at matching `}`
- Inside view: `<`, `>`, `/`, tag names, `=`, `{expr}`, string literals, text content

### 1.2 — Parser
- Parse standard Go portions as-is (pass through)
- Parse `view { ... }` block into an AST:
  - Elements: `<pkg.Component prop={expr}>children</pkg.Component>`
  - Self-closing: `<pkg.Component prop={expr} />`
  - Text content: raw text between tags
  - Expressions: `{goExpr}`
  - Control flow: `{if cond { ... }}`, `{for ... { ... }}`, `{switch ... { ... }}`
- Parse `type Props struct { ... }` — extract prop definitions
- Parse `type State struct { ... }` — extract state fields
- Detect lifecycle functions: `mount()`, `unmount()`

### 1.3 — AST (Intermediate Representation)
```
GoxFile
├── Package
├── Imports[]
├── Props (struct fields)
├── State (struct fields)
├── Lifecycle (mount, unmount)
├── Functions[] (helper funcs)
├── Vars[] (styles, etc.)
└── ViewBlock
    └── Node[]
        ├── Element{Tag, Props, Children}
        ├── TextNode{Content}
        ├── ExprNode{GoExpr}
        ├── IfNode{Cond, Body, Else}
        ├── ForNode{Clause, Body}
        └── SwitchNode{Expr, Cases}
```

### 1.4 — Code Generator (AST → .go)
- Emit package declaration
- Emit imports (add runtime imports as needed)
- Emit Props struct as-is
- Transform State struct: add reactive wrapper + setter methods
- Emit lifecycle registration (mount → runtime hook)
- Transform view block → Go `render()` function:
  - `<gox.View style={s}>` → `gox.Render("View", gox.Props{"style": s}, ...children)`
  - `{if cond { ... }}` → standard Go if wrapping render calls
  - `{for _, x := range items { ... }}` → standard Go for wrapping render calls
  - Text content → `gox.Text("string")`
  - `{expr}` → `gox.Expr(expr)`

### 1.5 — Integration Tests
- Parse sample .gox files from the spec
- Verify emitted .go files compile with `go build`
- Round-trip tests: parse → emit → parse again

## Phase 2: Runtime Library (gox package)
**Goal:** Provide the Go types and functions that generated code calls.

### 2.1 — Core Types
- `Style` struct (all CSS-like properties from spec)
- `Styles` map type
- `Merge()`, `When()` utility functions
- `Props` map type for internal render tree
- `Component` interface

### 2.2 — Render Tree
- Virtual node tree that represents the UI
- Diffing algorithm: old tree vs new tree → minimal update set
- State mutation → schedule re-render → diff → apply

### 2.3 — State Management
- Reactive state wrapper generated by compiler
- State mutations from any goroutine → marshal to UI thread
- Batch rapid mutations into single re-render

### 2.4 — Lifecycle Management
- Component mount/unmount hooks
- Component identity tracking (key prop)

### 2.5 — Navigation
- Router, Route, TabRouter from spec
- Navigate(), GoBack(), Replace() etc.
- Route param extraction → Props population

### 2.6 — HTTP Client (gox/http)
- Generic typed HTTP client from spec
- JSON serialization
- Base URL, default headers, timeout config

---

## Phase 3: Platform Bridge (iOS-first)
**Goal:** Connect the render tree to actual native views.

### 3.1 — Go ↔ Objective-C/Swift Bridge
- Use cgo + Objective-C runtime to call UIKit
- Render tree node → UIView subclass mapping
- Event callbacks: native → Go (button press, text change, scroll)

### 3.2 — Native View Mapping
- `gox.View` → UIView
- `gox.Text` → UILabel
- `gox.TextInput` → UITextField
- `gox.Button` → UIButton
- `gox.ScrollView` → UIScrollView
- `gox.Image` → UIImageView
- `gox.FlatList` → UICollectionView
- `gox.Modal` → presented UIViewController
- `gox.Switch` → UISwitch
- `gox.Slider` → UISlider

### 3.3 — Layout Engine
- Flexbox-style layout (Style properties → constraints)
- Options: use Yoga (C library) or implement subset
- Safe area handling

### 3.4 — Animation Bridge
- `gox.Animated` → Core Animation / UIView.animate
- Property interpolation on the native side

### 3.5 — Platform APIs
- `Platform()`, `OpenURL()`, `Haptic()`, etc.
- Storage: `Store()`/`Load()` → UserDefaults
- Clipboard, status bar, keyboard dismiss

## Phase 4: CLI Tool
**Goal:** Expo-like developer experience.

### 4.1 — `gox init <name>`
- Scaffold new project (go.mod, app/, screens/, components/)
- Generate starter app.gox, app.go, home screen

### 4.2 — `gox generate` (continuous)
- Walk project, find all .gox files
- Run compiler: .gox → .go (into `_gen/` or alongside)
- Watch mode: re-generate on file change

### 4.3 — `gox run ios`
- Generate native iOS project (Xcode project, storyboards, etc.)
- Compile Go → iOS binary (gomobile or direct cross-compile)
- Build and launch on simulator/device
- Config-driven: read gox.yaml/gox.go for bundle ID, Info.plist entries, etc.

### 4.4 — `gox run android` (future)
- Same pattern for Android

### 4.5 — `gox build`
- Production build: optimize, strip debug, produce .ipa/.apk

### 4.6 — Config System
```yaml
# gox.yaml
app:
  name: "MyApp"
  bundle_id: "com.example.myapp"
  version: "1.0.0"

ios:
  deployment_target: "16.0"
  info_plist:
    NSCameraUsageDescription: "We need camera access"
    UILaunchScreen: {}
  capabilities:
    - push-notifications
    - background-fetch

android:
  min_sdk: 26
  target_sdk: 34
  permissions:
    - android.permission.CAMERA
    - android.permission.INTERNET
```

Or alternatively as Go (gox_config.go):
```go
package main

import "gox/config"

var Config = config.App{
    Name:     "MyApp",
    BundleID: "com.example.myapp",
    Version:  "1.0.0",
    IOS: config.IOS{
        DeploymentTarget: "16.0",
        InfoPlist: map[string]any{
            "NSCameraUsageDescription": "We need camera access",
        },
    },
}
```

## Phase 5: Developer Experience
**Goal:** Fast iteration cycle.

### 5.1 — File Watcher
- Watch .gox files → auto-regenerate .go
- Watch .go files → auto-rebuild

### 5.2 — Hot Reload (stretch)
- Rebuild on change → re-deploy to simulator
- Preserve navigation state if possible

### 5.3 — Error Reporting
- Source-map .gox line numbers in compiler errors
- Clear error messages for common .gox syntax mistakes

### 5.4 — LSP Support (stretch)
- Syntax highlighting for .gox files
- Autocomplete for gox.* views and props
- Go-to-definition for components

## Implementation Order (What We Build First)

We start with **Phase 1** (compiler) because:
1. Everything depends on it
2. We can test it in isolation (input .gox → output .go)
3. It validates the language design — we'll discover spec issues early

Within Phase 1, the order is:
1. **Lexer** — tokenize .gox files (1.1)
2. **AST types** — define the IR (1.3)
3. **Parser** — tokens → AST (1.2)
4. **Code generator** — AST → .go (1.4)
5. **Tests** — validate with spec examples (1.5)

### First concrete task: Build the Lexer

The lexer needs to handle two modes:
- **Go mode**: Everything outside `view { }` — we don't need to fully tokenize Go, we just need to identify and pass through Go code, recognizing key patterns (package, import, type Props/State struct, func mount/unmount, var, and the `view` keyword)
- **View mode**: Inside `view { }` — tokenize JSX-like syntax

## Milestone 0: Hello World on iOS

Before building out the full system, prove feasibility with the simplest possible path:

**Target:** A .gox file with `View` + `Text` → compiled → running on iOS simulator showing "Hello World"

```gox
package app

import "gox"

view {
    <gox.View>
        <gox.Text>Hello World</gox.Text>
    </gox.View>
}
```

This touches every layer minimally:
1. Compiler: parse this simple .gox → emit .go
2. Runtime: just View + Text node types, no state/props/lifecycle yet
3. Bridge: View → UIView, Text → UILabel, that's it
4. CLI: just enough to compile and run on simulator

**We do NOT build yet:**
- Complex views (FlatList, ScrollView, Modal, etc.) — add one by one later
- State/Props/lifecycle — add after hello world works
- Navigation — add after basic components work
- Animations — much later
- Android — iOS first, Android later

## Spec Improvements Identified

1. **Missing: CLI commands** — Need to define `gox init`, `gox run`, `gox build`, `gox dev`
2. **Missing: Config system** — How users configure native settings without touching ios/android folders
3. **Missing: Platform bridge details** — How Go actually calls UIKit at runtime
4. **Missing: Generated file layout** — Where do .go files from .gox go? Alongside? Separate dir?
5. **Missing: Entry point** — How does the app actually start? main.go? auto-generated?
6. **Missing: Error model** — What happens when .gox syntax is wrong? Error format?
7. **go.mod says go 1.25.0** — This doesn't exist yet. Should be go 1.24 or whatever is current.
8. **ActivityIndicator** — Used in spinner example but not listed in primitive views table
9. **FlatList generics in JSX** — How do you express `FlatList[Post]` in JSX syntax? Not covered.
10. **Width: -1 in example** — `styles["postImage"]` uses Width: -1 which presumably means "100%" — need to document this convention
11. **State field access** — Direct field access (`loading = true`) requires compiler magic. Need to specify scoping rules: what if a local var shadows a state field?
12. **Component references** — `home.Screen`, `posts.Screen` — what is `.Screen`? Not defined. Is it auto-generated?
